# TMDb API Executor (Worker)
**Teddy** — это изолированный воркер, чья единственная задача: забирать подготовленные задачи из Redis и корректно доставлять их в TMDb API, соблюдая технические лимиты.
## 1. Философия "Тупого исполнителя"
Teddy не обладает бизнес-логикой и не имеет прямого доступа к основному хранилищу пользователей для получения секретов.
 * **Отсутствие инициативы**: Teddy не решает, кого синхронизировать. Он ждет задачу от **Fletcher** (Центра).
 * **Контекст в задаче**: Все необходимые данные (Access Token, Payload, Endpoint) приходят в теле задачи из Redis.
 * **Слепая вера**: Teddy не проверяет валидность токена перед отправкой — он выполняет запрос и возвращает результат Fletcher'у.
## 2. Формат задачи из Redis (Input)
Fletcher помещает в Redis JSON-объект, который Teddy парсит и исполняет:
```json
{
  "task_id": "uuid-12345",
  "method": "POST",
  "endpoint": "/4/list/{list_id}/items",
  "access_token": "DECRYPTED_TOKEN_FROM_FLETCHER",
  "payload": { ... },
  "priority": 1
}

```
## 3. Обязанности Teddy (Responsibility)
### А. Соблюдение Rate Limits (Throttling)
Это главная интеллектуальная нагрузка Teddy. Он следит за тем, чтобы запросы к TMDb не превышали установленные пороги:
 * **GET**: ~10 запросов/сек.
 * **POST/DELETE**: ~2 запроса/сек.
 * **Backoff**: При получении 429 Too Many Requests, Teddy обязан приостановить обработку очереди для этого типа задач и сигнализировать Fletcher'у о перегрузке.
### Б. Выполнение сетевых операций
 * Использование aiohttp с пулом соединений (Keep-Alive) для минимизации задержек на Handshake.
 * Установка таймаутов на каждый запрос (не более 10-15 секунд).
### В. Обратная связь (Output)
После выполнения запроса Teddy кладет результат в соответствующий ключ ответа в Redis или отправляет статус через Pub/Sub:
 * **Success**: Код 200/201 + Body.
 * **Retry**: Код 5xx или Network Timeout (Fletcher решит, переотправлять или нет).
 * **Fail**: Код 401 (Token Expired) или 404.
## 4. Взаимодействие с Fletcher
Fletcher выступает в роли диспетчера (Orchestrator):
 1. **Мониторинг очереди**: Fletcher следит за длиной очереди в Redis.
 2. **Flow Control**: Если Teddy не успевает разгребать задачи (очередь растет), Fletcher замедляет генерацию новых задач на синхронизацию, чтобы не перегружать воркера и API.
 3. **Security**: Fletcher достает зашифрованный токен из БД, расшифровывает его и передает Teddy в «чистом» виде только на время выполнения задачи.
## 5. Исключения
Если Teddy ловит критическую ошибку сети или API, он не пытается «лечить» систему. Он возвращает ошибку Fletcher'у и переходит к следующей задаче. Teddy — расходный материал, Fletcher — контролирующий орган.
